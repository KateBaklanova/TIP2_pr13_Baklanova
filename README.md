## Цели работы

- Освоить базовую работу с брокером сообщений RabbitMQ в приложении на Go, научиться публиковать сообщения в очередь, реализовывать отдельного потребителя сообщений, подтверждать успешную обработку через ack и понимать назначение очередей сообщений в микросервисной архитектуре.

## Задачи

- Изучить назначение брокера сообщений и понять, зачем он используется в распределённых backend-системах.
- Научиться локально поднимать RabbitMQ через docker-compose.
- Освоить публикацию сообщений из Go-приложения в очередь RabbitMQ.
- Реализовать отдельный consumer, который читает сообщения и подтверждает обработку.
- Научиться использовать ack и prefetch в базовом сценарии обработки сообщений.
- Подготовить работающий учебный пример обмена сообщениями между сервисом tasks и worker-процессом.

### Содержание проекта

**Auth service** (порт 8081 +  gRPC порт 50051)
- Аутентификация пользователей
- Выдача токенов
- Проверка токенов через gRPC

**Tasks service** (порт 8082)
- CRUD для задач (TODO-список)
- Перед каждой операцией проверяет токен, отправляя gRPC-запрос в Auth service

**Worker service** (отдельный процесс)
- Подключение к RabbitMQ
- Чтение сообщений из очереди task_events
- Обработка событий task.created
- Подтверждение обработки через ack
- Логирование полученных событий

### Контрольные вопросы

1. Зачем нужен брокер сообщений, если между сервисами можно использовать HTTP?

Для асинхронной обработки задач, не требующих мгновенного ответа, и для развязки сервисов

2. Что такое producer и consumer?

Producer - публикует сообщения в очередь, consumer - читает и обрабатывает сообщения из очереди.

3. Почему сообщение нужно публиковать только после успешного создания задачи?

Чтобы не было события о задаче, которой не существует в системе

4.Что такое ack и зачем он нужен?

Подтверждение обработки сообщения, чтобы брокер удалил его из очереди

5. Почему возможна повторная доставка сообщения?

Если consumer не отправил ack до сбоя или разрыва соединения, брокер переотправляет сообщение

6. Что делает параметр prefetch?

Ограничивает количество неподтверждённых сообщений, которые брокер может отправить одному consumer

6. Чем durable queue отличается от non-durable queue?

Durable очередь переживает перезапуск брокера, non-durable удаляется при его остановке.

7. Почему JSON удобен как учебный формат сообщения?

Понятен визуально, легко сериализуется и десериализуется в go

8. Что означает режим best effort при публикации события?

Ошибка публикации только логируется, задача всё равно считается созданной, клиент получает успешный ответ

9. Почему worker лучше выносить в отдельный процесс, а не смешивать с HTTP-обработчиками?

Чтобы не блокировать основной HTTP-сервер обработкой сообщений и обеспечить независимое масштабирование
